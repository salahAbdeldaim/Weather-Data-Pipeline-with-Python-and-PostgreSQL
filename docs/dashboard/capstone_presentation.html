Automated extraction, transforming, validation, and analytical presentation of high-resolution atmospheric telemetry across 260 Egyptian cities and governorates.

Problem Statement & Motivation

🧐 Why Did We Build This Platform?

⚠️ Existing Challenges (`The Problem`)

  • Scattered & Manual Telemetry: Traditional weather data across Egyptian governorates is fragmented, unverified, and requires manual scraping.
  • No Historical Time-Series Tracking: Standard weather APIs only provide instant snapshots, losing vital day-over-day temperature shift trends.
  • Lack of Data Quality Guardrails: Raw external JSON APIs frequently inject corrupt data types, missing coordinates, or extreme null values.
  • Closed Silos: Decision-makers in agriculture, aviation, and logistics lack instant, interactive, publicly embeddable analytical BI endpoints.

✅ Our Engineering Solution

  • 100% Automated Ingestion: Self-healing ETL pipeline pulling from Open-Meteo API across 260 accurate latitude/longitude coordinates.
  • Dual-Storage Architecture: Preserving full historical time-series (`874+ records`) while serving instantaneous real-time views (`260 exact cities`).
  • Strict Pydantic Guardrails: Every incoming payload undergoes schema validation before hitting the relational database.
  • Open Data Platform: Public UUID question endpoints enabling zero-credential iframe embeds across external portals.

Reliability & Data Integrity

🛡️ Idempotent Upsert & Data Guardrails

🔐 Idempotent Upsert (`ON CONFLICT DO UPDATE`)

Why didn't we just use standard INSERT INTO or truncate the table on every pipeline run?

  • Zero Duplication on Retries: In enterprise Data Engineering, pipelines may re-run due to network hiccups. Our table constraint UNIQUE(city, ingestion_time) paired with ON CONFLICT DO UPDATE guarantees exact idempotency.
  • Time-Series Preservation vs Snapshot: Our main table stores cumulative history (`874+ rows`), allowing trend analysis. Meanwhile, our database view view_latest_weather_per_city delivers exact 260-city current snapshots!

📐 Pydantic Schema Guardrails

Before any API payload touches our database cursor, it must pass strict runtime schema enforcement:

  • Data Type Safety: Automatic coercion and verification that temperature, humidity, and wind speed are valid floats.
  • Null / Anomaly Rejection: Prevents missing coordinates (`latitude/longitude`) or malformed weather codes from contaminating the analytical schema.
  • Decoupled Transformation: Separates extraction anomalies from load failures, ensuring high pipeline availability.
Orchestration & Observability

⚡ n8n DataOps Workflow & Telegram Telemetry

⚙️ Why n8n Cloud Orchestration?

Rather than relying on unmonitored raw crontabs, we engineered a visual, cloud-hosted Directed Acyclic Graph (`DAG`) workflow:

  • Daily Cron Trigger: Automatically wakes up and triggers the Docker ETL container execution daily at scheduled UTC windows.
  • Database Health Verification: Executes automated SQL validation queries after extraction to verify row counts and data freshness.
  • Self-Healing Execution: Handles timeout exceptions and logs execution metrics seamlessly.
